## 摘要

DeepSeek-V4 系列是 DeepSeek 推出的第四代大语言模型，包含两个版本：**DeepSeek-V4-Pro**（1.6T总参数，49B激活参数）和 **DeepSeek-V4-Flash**（284B总参数，13B激活参数）。两者均支持 **100万token** 的上下文长度，在保持强大性能的同时实现了极致的长上下文效率。

### 核心性能指标对比

| 指标 | DeepSeek-V3.2 | DeepSeek-V4-Pro | DeepSeek-V4-Flash |
|------|---------------|-----------------|-------------------|
| 总参数量 | 671B | 1.6T | 284B |
| 激活参数量 | 37B | 49B | 13B |
| 上下文长度 | 128K | **1M** | **1M** |
| 1M上下文推理FLOPs | 基准 | **27%** | **10%** |
| 1M上下文KV Cache | 基准 | **10%** | **7%** |

## 一、架构创新

### 1.1 混合注意力架构（Hybrid Attention Architecture）

DeepSeek-V4 的核心架构创新是引入了**混合注意力机制**，结合两种高效注意力模块：

#### 1.1.1 压缩稀疏注意力（Compressed Sparse Attention, CSA）

**核心思想**：将KV缓存沿序列维度压缩，然后应用稀疏注意力

**技术细节**：
- **Token级压缩器**：将每 `m` 个token的KV条目压缩为1个条目，压缩序列长度至 1/m
- **双重压缩权重**：使用两组KV条目（`Ca`, `Cb`）和压缩权重（`Za`, `Zb`）进行软压缩
- **重叠压缩策略**：每个压缩条目由 2m 个KV条目派生，相邻压缩块之间有重叠
- **Lightning Indexer**：低秩索引器用于稀疏选择Top-k压缩KV条目
- **共享KV MQA**：采用Multi-Query Attention，压缩后的KV条目同时作为Key和Value
- **分组输出投影**：将输出分成 g 组进行投影，降低计算负担

```
CSA架构流程：
Input → Token-Level Compressor → Compressed KV Entries → 
Top-k Selector → Sparse Attention → Grouped Output Projection → Output
```

**配置参数**（V4-Pro）：
- 压缩率 `m = 4`
- 索引器Query头数 `n_h^I = 64`
- 索引器头维度 `c_I = 128`
- 稀疏注意力Top-k = 1024

#### 1.1.2 重度压缩注意力（Heavily Compressed Attention, HCA）

**核心思想**：更激进的KV缓存压缩，不进行稀疏注意力

**技术细节**：
- 使用更大的压缩率 `m'`（≫ m），将每 `m'` 个token压缩为1个条目
- **无重叠压缩**：与CSA不同，HCA不进行重叠压缩
- 保留密集注意力计算，但作用于压缩后的KV条目
- 计算复杂度更低，适合处理极长序列

**配置参数**（V4-Pro）：
- 压缩率 `m' = 128`

#### 1.1.3 CSA与HCA的混合配置

**层间交替策略**：
- **前2层**：使用纯滑动窗口注意力（建立局部依赖）
- **后续层**：CSA和HCA交替使用
- **V4-Pro**：61层Transformer，HCA用于关键层
- **V4-Flash**：43层Transformer，前2层纯滑动窗口

**效率提升效果**（1M上下文 vs V3.2）：
- V4-Pro：推理FLOPs降低至 **27%**，KV Cache降低至 **10%**
- V4-Flash：推理FLOPs降低至 **10%**，KV Cache降低至 **7%**

### 1.2 流形约束超连接（Manifold-Constrained Hyper-Connections, mHC）

**背景**：标准超连接（Hyper-Connections, HC）通过扩展残差流宽度增强模型，但多层堆叠时容易出现数值不稳定。

**核心创新**：

#### 1.2.1 双随机矩阵约束

将残差映射矩阵 `B_l` 约束到**双随机矩阵流形**（Birkhoff多面体）：

```
M = {M ∈ R^(n×n) | M·1_n = 1_n, 1_n^T·M = 1_n^T, M ≥ 0}
```

**数学特性**：
- 谱范数 ‖B_l‖₂ ≤ 1，确保非扩张性变换
- 矩阵乘法封闭性，保证深层堆叠稳定性
- 通过 **Sinkhorn-Knopp算法** 进行投影（20次迭代）

#### 1.2.2 动态参数生成

mHC的三个线性映射参数（输入映射 `A_l`、残差变换 `B_l`、输出映射 `C_l`）都是动态生成的：

```
原始参数 = α · (输入 · W) + S
```

其中：
- `W`：可学习的静态参数
- `S`：可学习的静态偏置
- `α`：可学习的门控因子（小值初始化）

#### 1.2.3 参数约束应用

- **输入/输出映射**：使用Sigmoid函数 σ(·) 确保非负性和有界性
  - `A_l = σ(Ã_l)`
  - `C_l = 2σ(Č_l)`
- **残差映射**：通过Sinkhorn-Knopp投影到双随机矩阵

**效率优化**：
- 融合内核实现训练和推理
- 选择性检查点策略（重计算中间状态）
- 与DualPipe 1F1B重叠方案结合
- **总体开销**：仅占重叠1F1B流水线阶段的 **6.7%**

### 1.3 Muon优化器

**来源**：基于 Jordan et al. (2024) 和 Liu et al. (2025) 的Muon优化器改进

#### 1.3.1 核心算法

```python
# Muon优化器伪代码
for each training step t:
    for each weight matrix W ∈ R^(n×m):
        G_t = ∇_W L_t(W_{t-1})           # 计算梯度
        M_t = μ·M_{t-1} + G_t              # 动量累积
        O'_t = HybridNewtonSchulz(μ·M_t + G_t)  # 正交化
        O_t = O'_t · √(max(n,m)) · γ       # 缩放更新RMS
        W_t = W_{t-1}·(1-ηλ) - η·O_t      # 权重衰减和更新
```

#### 1.3.2 混合Newton-Schulz迭代

用于近似矩阵正交化（计算 UV^T）：

**两阶段迭代策略**：
1. **前8步**：快速收敛系数 `(a, b, c) = (3.4445, -4.7750, 2.0315)`
2. **后2步**：稳定收敛系数 `(a, b, c) = (2, -1.5, 0.5)`

迭代公式：
```
M_k = a·M_{k-1} + b·(M_{k-1}·M_{k-1}^T)·M_{k-1} + c·(M_{k-1}·M_{k-1}^T)^2·M_{k-1}
```

#### 1.3.3 工程优化

**混合ZeRO策略**：
- Muon需要完整梯度矩阵，与ZeRO冲突
- 对密集参数：限制最大ZeRO并行度，使用背包算法分配
- 对MoE参数：独立优化每个专家，展平所有投影矩阵

**通信优化**：
- BF16矩阵乘法进行Newton-Schulz迭代
- 梯度随机量化到BF16，通信量减半
- 两阶段归约策略（all-to-all + 本地求和）保持数值稳健

## 二、基础设施创新

### 2.1 细粒度专家并行通信-计算重叠

**核心问题**：专家并行（EP）需要复杂的节点间通信，对互联带宽和延迟要求高。

#### 2.1.1 关键洞察

MoE层可分解为四个阶段：
1. **Dispatch**（通信受限）：token分发到专家
2. **Linear-1**（计算受限）：第一层线性投影
3. **激活函数**（计算受限）：SwiGLU + 类型转换
4. **Linear-2**（计算受限）：第二层线性投影
5. **Combine**（通信受限）：结果聚合

**发现**：单次MoE层中，总通信时间 < 总计算时间 → 通信可被计算隐藏

#### 2.1.2 细粒度专家分区方案

**波次（Wave）调度策略**：
- 将专家划分为多个波次（每个波次包含一小部分专家）
- 波次内完成通信后立即开始计算，无需等待其他波次
- 稳态时：当前波次计算、下一波次通信、已完成波次发送结果 **三者并行**

**性能提升**（相比非融合基线）：
- 通用推理工作负载：**1.50 ~ 1.73×**
- RL rollout等延迟敏感场景：**最高1.96×**

**开源实现**：**MegaMoE**（作为DeepGEMM组件）

#### 2.1.3 硬件设计建议

基于实际部署经验，提出四点硬件设计建议：

1. **计算-通信比例**：C/B ≤ 2d = 6144 FLOPs/Byte
   - 每GBps互联带宽可隐藏6.1 TFLOP/s计算

2. **功率预算**：极端内核融合同时推高计算、内存和网络负载，建议提供充足功率余量

3. **通信原语**：采用pull-based方法（GPU主动从远程读取），避免细粒度push的高通知延迟

4. **激活函数**：建议用低成本逐元素激活替代SwiGLU（无指数或除法），增大中间维度

### 2.2 TileLang DSL内核开发

**背景**：复杂架构会导致数百个细粒度Torch ATen算子。

**解决方案**：采用 **TileLang**（领域特定语言）开发融合内核。

#### 2.2.1 主机代码生成（Host Codegen）

**问题**：高性能内核的CPU端编排开销日益突出，Python实现的运行时检查带来固定开销。

**解决方案**：
- 在IR级别协同生成设备内核和轻量级主机启动器
- 嵌入元数据（数据类型、形状约束、步幅/布局假设）
- 基于TVM-FFI框架构建紧凑调用约定和零拷贝张量互操作
- **效果**：CPU端验证开销从数十/数百微秒降至 **<1微秒**

#### 2.2.2 SMT求解器辅助整数分析

**集成Z3 SMT求解器**到TileLang代数系统：

- 将整数表达式翻译为Z3的**无量词非线性整数算术（QF_NIA）**
- 基于ILP求解器处理标准线性整数表达式
- 非线性推理能力应对向量化等高级挑战
- **编译时间开销**：仅几秒
- **影响**：向量化、屏障插入、代码简化等多个环节

#### 2.2.3 数值精度与位级可复现性

**精度保证**：
- 默认禁用fast-math优化
- 精度影响近似仅作为显式可选操作符（`T.__exp`, `T.__log`, `T.__sin`）
- 提供IEEE-754兼容内联函数（`T.ieee_fsqrt`, `T.ieee_fdiv`, `T.ieee_add`）

**位级可复现性**：
- 对齐TileLang代数简化和降低规则与主流CUDA工具链（NVCC）
- 布局注解（`T.annotate_layout`）固定布局相关降低决策
- 保持求值和累加顺序与参考CUDA实现一致

### 2.3 高性能批不变和确定性内核库

#### 2.3.1 批不变性（Batch Invariance）

确保任何给定token的输出位级相同，与其在批次中的位置无关。

**注意力内核**：
- 不使用split-KV（负载均衡但破坏批不变性）
- 双内核策略：
  - **内核1**：单SM内计算整个序列的注意力输出（高吞吐）
  - **内核2**：多SM处理最后部分填充的波次（低延迟）
- 内核2的累加顺序与内核1一致
- 使用分布式共享内存实现SM间高速数据交换

**矩阵乘法**：
- 端到端替换cuBLAS为DeepGEMM
- 放弃split-k技术（虽然性能更好但破坏批不变性）
- 引入优化集使实现性能匹配甚至超越标准split-k

#### 2.3.2 确定性（Determinism）

对调试硬件/软件问题、定位损失异常至关重要。

**来源**：非确定性通常来自非确定性累加顺序（原子加法指令）

**解决方案**：
- **注意力反向**：为每个SM分配独立累加缓冲区，全局确定性求和
- **MoE反向**：单rank内token排序预处理 + 多rank间缓冲区隔离
- **mHC矩阵乘法**：小batch必须用split-k时，输出每个split部分，后续内核确定性归约

### 2.4 FP4量化感知训练（QAT）

**应用场景**：MoE专家权重和CSA索引器QK路径

#### 2.4.1 训练阶段模拟

- FP32主权重 → FP4量化 → FP8反量化 → 计算
- **关键**：FP4到FP8反量化是无损的（FP8 E4M3比FP4 E2M1多2个指数位）
- 反向传播梯度相对于FP8权重计算，直接传播回FP32主权重
- 等价于通过量化操作应用直通估计器（STE）

#### 2.4.2 推理阶段直接量化

- RL推理和部署时直接使用FP4量化权重
- 确保采样行为与在线部署完全一致
- 显著降低内存占用和采样延迟

#### 2.4.3 索引分数量化

- 将索引分数 `I` 从FP32量化到BF16
- Top-k选择器加速 **2×**
- KV条目召回率保持 **99.7%**

## 三、训练创新

### 3.1 数据构建

**训练数据规模**：
- V4-Flash：32T tokens
- V4-Pro：33T tokens

**数据改进**：
- 持续优化数据处理流水线
- 网页数据过滤：去除批量自动生成和模板化内容（防止模型崩溃）
- 数学和编程语料仍是核心，**中期训练阶段注入Agent数据**
- 多语言数据扩展：涵盖英语、中文外更多语言
- **重点**：长文档数据策划（科学论文、技术报告等）

**分词器**：
- 基于V3分词器，添加特殊token用于上下文构建
- 词汇表保持128K
- 保留token-splitting和Fill-in-Middle（FIM）策略

**训练技巧**：
- 文档打包：将不同来源文档打包成适当序列，最小化样本截断
- 样本级注意力掩码（与V3不同）

### 3.2 训练稳定性解决方案

训练万亿参数MoE模型面临显著稳定性挑战。

#### 3.2.1 预判路由（Anticipatory Routing）

**问题**：骨干网络和路由网络同步更新导致训练不稳定。

**解决方案**：
- 在步骤 `t` 使用当前参数 `θ_t` 计算特征
- 但路由索引使用历史参数 `θ_{t-Δt}` 计算和应用

**工程优化**：
1. **预计算策略**：在步骤 `t-Δt` 提前获取步骤 `t` 的数据，预判计算并缓存路由索引
2. **开销控制**：流水线执行和EP通信重叠，额外wall-clock时间开销仅约 **20%**
3. **动态触发**：自动检测损失异常 → 触发短回滚 → 激活预判路由 → 运行一段时间后恢复标准训练

**效果**：以可忽略的总体额外训练开销避免损失异常，不损害模型性能。

#### 3.2.2 SwiGLU裁剪（SwiGLU Clamping）

**配置**：
- SwiGLU线性组件限制在 `[-10, 10]`
- 门组件上限为 `10`

**效果**：有效消除异常值，显著稳定训练过程，不损害性能。

### 3.3 序列长度渐进扩展

**四阶段训练**：
1. **4K**（初始阶段，密集注意力预热1T tokens）
2. **16K**
3. **64K**（引入稀疏注意力）
4. **1M**（最终目标）

**稀疏注意力引入策略**：
- 先用密集注意力预热
- 64K长度时引入稀疏注意力
- 先短暂预热Lightning Indexer
- 然后大部分时间使用稀疏注意力训练

## 四、后训练创新

### 4.1 两阶段后训练范式

**传统方法**：预训练 → 微调 → 混合RL

**DeepSeek-V4新方法**：
1. **专家特训阶段**：为每个领域（数学、代码、Agent、指令遵循）独立训练专家模型
   - SFT建立基础能力
   - GRPO强化学习优化领域对齐行为
2. **策略蒸馏阶段**：通过**在线策略蒸馏（OPD）**将多个专家能力融合到统一模型

### 4.2 推理努力度模式（Reasoning Effort Modes）

三个推理模式：

| 模式 | 特点 | 典型用例 | 上下文窗口 |
|------|------|----------|------------|
| **Non-think** | 快速、直觉响应 | 日常任务、紧急反应、低风险决策 | 8K |
| **Think High** | 有意识逻辑分析，较慢但更准 | 复杂问题解决、规划、中风险决策 | 128K |
| **Think Max** | 推理到极致，慢但强 | 探索模型推理能力边界 | 384K |

**Think Max特殊配置**：
- 系统提示注入特定指令：
  ```
  "推理努力度：绝对最大值，不允许任何捷径。
  你必须非常彻底地思考，全面分解问题以解决根本原因，
  严格对照所有潜在路径、边界情况和对抗场景压力测试你的逻辑。
  明确写出你的整个思考过程..."
  ```

### 4.3 生成式奖励模型（Generative Reward Model, GRM）

**传统方法**：人工标注训练标量奖励模型（RLHF）。

**DeepSeek-V4创新**：
- 对难以验证的任务，策划评分标准引导的RL数据
- 使用**生成式奖励模型**评估策略轨迹
- **关键**：直接对GRM应用RL优化
- 演员网络原生作为GRM，联合优化模型评估能力和标准生成能力
- 统一角色使内部推理能力融合到评估过程中
- 仅需少量多样化人工标注，模型逻辑可泛化到复杂任务

### 4.4 工具调用Schema与特殊Token

**工具调用格式**：
- 使用特殊 `"|DSML|"` token
- XML格式进行工具调用
- 有效缓解转义失败，降低工具调用错误

**交错思考（Interleaved Thinking）**：

V3.2策略：工具结果轮保留推理痕迹，新用户消息时丢弃。

V4改进（利用1M上下文窗口）：
- **工具调用场景**：完整保留所有推理历史（跨用户消息边界）
- **普通对话场景**：新用户消息时丢弃之前推理内容

### 4.5 快速指令（Quick Instruction）

**问题**：聊天机器人场景中，生成响应前需执行辅助任务（是否触发搜索、意图识别等），传统方法用独立小模型，导致冗余预填充。

**解决方案**：
- 直接向输入序列追加专用特殊token
- 每个token对应特定辅助任务
- 复用已计算的KV缓存，完全避免冗余预填充
- 某些任务（生成搜索查询、确定权威性和领域）可并行执行
- **效果**：显著降低用户感知的首token时间（TTFT）

支持的Quick Instruction token：
- `<|action|>`：判断是否需网络搜索
- `<|title|>`：生成对话标题
- `<|query|>`：生成搜索查询
- `<|authority|>`：分类权威性需求
- `<|domain|>`：识别领域
- `<|extracted_url|>`, `<|read_url|>`：判断URL是否需获取

### 4.6 全词汇在线策略蒸馏（Full-Vocabulary OPD）

**OPD目标函数**：

```
L_OPD(θ) = Σ_{i=1}^N w_i · D_KL(π_θ || π_Ei)
```

其中 `π_Ei` 是第i个专家教师模型，从学生 `π_θ` 采样保持在线学习。

**传统简化方法的问题**：
- 将全词汇KL损失简化为token级KL估计
- 高方差梯度估计，常导致训练不稳定

**DeepSeek-V4方案**：
- 采用**全词汇logit蒸馏**
- 保留完整logit分布计算反向KL损失
- 更稳定梯度估计，忠实蒸馏教师知识

**工程实现**：
- 所有教师权重卸载到集中式分布式存储
- 前向传播时按需加载（ZeRO-like分片）
- 仅缓存教师最后一层隐藏状态（非完整logits）
- 训练时通过预测头重建logits
- 样本按教师索引排序，确保每批最多一个教师头在显存
- 异步加载/卸载，不阻塞关键路径计算
- 专用TileLang内核加速KL散度计算

### 4.7 可抢占和容错Rollout服务

**背景**：GPU集群采用集群级可抢占任务调度，硬件故障频发。

**解决方案**：

**Token级预写日志（WAL）**：
- 每个生成请求维护独立WAL
- 每生成新token立即追加到WAL
- 抢占时暂停推理引擎，保存未完成请求的KV缓存
- 恢复时使用持久化WAL和KV缓存继续解码
- 致命硬件错误时，用WAL中token重新预填充重建KV缓存

**数学正确性**：
- 不能从头重新生成未完成请求（引入长度偏差）
- 短响应更可能在中断中存活
- 从头生成会使模型倾向于产生更短序列
- 使用一致种子重新生成虽可解决正确性问题，但效率远低于WAL方法

## 五、Agent能力基础设施

### 5.1 DeepSeek弹性计算（DSec）沙盒平台

**组成**：三个Rust组件
1. **API网关（Apiserver）**
2. **每主机代理（Edge）**
3. **集群监控（Watcher）**

**互联**：自定义RPC协议，基于3FS分布式文件系统水平扩展

**规模**：单个DSec集群管理**数十万个并发沙盒实例**

### 5.2 四种执行底层

统一Python SDK（libdsec）抽象四种执行底层：

| 底层 | 技术基础 | 特点 | 适用场景 |
|------|----------|------|----------|
| **Function Call** | 预 warmed 容器池 | 无冷启动开销 | 无状态快速调用 |
| **Container** | Docker + EROFS按需加载 | 完全Docker兼容 | 标准容器工作负载 |
| **microVM** | Firecracker | VM级隔离 | 安全敏感、高密度部署 |
| **fullVM** | QEMU | 任意客户操作系统 | 特殊OS需求 |

切换仅需修改参数。

### 5.3 快速镜像加载

**分层存储策略**：
- 容器：基础镜像和文件系统提交存储为3FS支持的只读EROFS层，直接挂载到overlay lowerdirs
- microVM：使用overlaybd磁盘格式，只读基础层在3FS上，写入到本地copy-on-write层
- 快照可链式组合，支持高效版本控制和毫秒级恢复

### 5.4 大规模并发下的密度优化

**两个资源瓶颈**：
1. **内存**：重复页面缓存占用、内存回收实现安全超售
2. **CPU**：容器运行时自旋锁竞争，降低每沙盒CPU开销

### 5.5 轨迹日志与抢占安全恢复

**全局有序轨迹日志**：持久化记录每个命令调用及其结果

**三大用途**：
1. **客户端快进**：训练任务被抢占时保留沙盒资源，恢复时回放缓存结果加速任务恢复，防止非幂等操作重复执行错误
2. **细粒度溯源**：每次状态变更的来源和结果可追溯
3. **确定性重放**：任何历史会话可从轨迹忠实重现

## 六、长上下文推理优化

### 6.1 推理框架

**异构KV缓存结构**：

DeepSeek-V4引入多种类型KV条目，大小和更新规则各异：
- Lightning Indexer引入额外维度
- CSA和HCA压缩技术将序列长度分别压缩至 1/m 和 1/m'
- 滑动窗口注意力（SWA）层有不同缓存大小和命中/驱逐策略
- 压缩分支中，每m个token生成一个KV条目，不足m时保留在缓冲区

**两层缓存架构**：
1. **传统KV缓存**：用于CSA/HCA
2. **状态缓存（State Cache）**：用于SWA和未准备好压缩的token

**状态缓存布局**：
- 每请求分配固定大小缓存块
- SWA段：存储最近 `n_win` 个token的KV条目
- CSA/HCA段：存储未压缩的尾部状态

**传统KV缓存**：
- 每请求分配多个块
- 每块覆盖 `lcm(m, m')` 个原始token
- 产生 `k1 = lcm(m,m')/m` 个CSA压缩token和 `k2 = lcm(m,m')/m'` 个HCA压缩token

### 6.2 磁盘KV缓存存储

**CSA/HCA压缩KV条目**：
- 直接存储所有压缩KV条目到磁盘
- 命中前缀时读取并复用，直到最后一个完整压缩块
- 尾部不完整块的token需重新计算恢复未压缩KV条目

**SWA KV条目（三种策略）**：

| 策略 | 存储内容 | 计算冗余 | 存储效率 | 适用场景 |
|------|----------|----------|----------|----------|
| **Full SWA Caching** | 所有token的完整SWA KV | 零 | 低（写密集型） | 计算优先 |
| **Periodic Checkpointing** | 每p个token的最后n_win个 | 可配置 | 中等 | 平衡型 |
| **Zero SWA Caching** | 不存储 | 需重计算n_win·L个token | 高 | 存储优先 |

**Zero SWA Caching重计算**：
- 利用缓存的CSA和HCA KV条目
- 每层SWA KV条目仅依赖前层最近n_win个token的SWA KV
- 重计算最后 `n_win · L` 个token即可恢复最后 `n_win` 个SWA KV条目

### 6.3 上下文并行优化

**传统CP挑战**：
- 训练样本由多序列打包，每序列独立压缩
- 压缩KV长度通常 < s/m，各rank不同
- 压缩需要m个连续KV条目，可能跨越CP rank边界

**两阶段通信方法**：
1. **第一阶段**：rank i发送最后m个未压缩KV条目给rank i+1
2. rank i+1将接收的条目与本地s个未压缩KV条目一起压缩，生成固定长度 `s/m + 1` 的压缩条目（含填充）
3. **第二阶段**：all-gather收集本地压缩KV条目
4. 融合选择-填充操作符重组完整压缩KV条目集，填充条目放在尾部

## 八、局限性与未来方向

### 8.1 当前局限性

1. **架构复杂度**：为追求极致长上下文效率，采用相对激进的架构设计，保留许多初步验证的组件，使架构较复杂。

2. **训练稳定性机制原理不清**：预判路由和SwiGLU裁剪虽有效，但底层原理尚未充分理解。

3. **V4-Pro API成本**：因高端算力受限，吞吐有限，API价格目前较高，预计下半年才能下降。

### 8.2 未来方向

1. **架构简化**：更全面和原则性地研究，将架构提炼至最本质设计，保持性能同时更优雅。

2. **训练稳定性研究**：深入研究训练稳定性基础问题，加强内部指标监控，建立更原则性和预测性的稳定训练方法。

3. **模型稀疏性探索**：
   - 探索更多维度的模型稀疏性（如更稀疏的嵌入模块）
   - 进一步提高计算和内存效率

4. **低延迟架构**：持续研究低延迟架构和系统技术，使长上下文部署和交互更响应。

5. **长时序Agent任务**：持续迭代和探索长时序、多轮Agentic任务。

6. **多模态能力**：将多模态能力整合到模型中。

7. **数据策划与合成**：开发更好的数据策划和合成策略，持续提升模型智能、鲁棒性和实用性。

## 九、总结

DeepSeek-V4 代表了开源大语言模型的重要里程碑，其核心贡献包括：

### 9.1 技术创新

1. **混合注意力架构**（CSA + HCA）：首次实现百万token上下文的高效处理，推理成本降低 **90%**+
2. **流形约束超连接**（mHC）：解决深层超连接数值不稳定问题，增强信号传播
3. **Muon优化器**：更快的收敛速度和更好的训练稳定性
4. **FP4 QAT**：在保持精度的同时大幅降低内存和计算需求
5. **预判路由和SwiGLU裁剪**：有效解决万亿参数MoE训练稳定性问题

### 9.2 工程创新

1. **细粒度EP通信-计算重叠**：MegaMoE实现1.73×+加速
2. **TileLang DSL**：平衡开发效率和运行时性能
3. **批不变和确定性内核**：确保训练可复现性
4. **DSec沙盒平台**：支持数十万个并发Agent实例
5. **Token级WAL**：实现可抢占和容错的RL rollout服务

### 9.3 训练范式创新

1. **两阶段后训练**：专家特训 + 在线策略蒸馏
2. **生成式奖励模型**：统一评估和生成能力
3. **交错思考机制**：充分利用百万token上下文进行Agent任务
4. **快速指令**：消除辅助任务冗余预填充

### 9.4 性能成就

- **知识能力**：开源模型新标杆，SimpleQA-Verified领先20个百分点
- **推理能力**：匹敌GPT-5.4，Codeforces排名第23位
- **Agent能力**：内部编码任务接近Claude Opus 4.5
- **长上下文**：1M token检索超越Gemini-3.1-Pro
- **效率**：V4-Flash以1/6参数在基础能力上超越V3.2

DeepSeek-V4 的发布标志着百万token长上下文时代的来临，为测试时扩展、长时序任务和在线学习等新范式奠定了必要基础。
